golang的sync.Map实现原理

发表于 2019-07-01 更新于 2024-07-09 分类于 1-编程语言 ， golang ， 1_golang基础 阅读次数： Valine：
本文字数： 6.1k 阅读时长 ≈ 6 分钟

map的读写删除都不是原子操作，因此需要控制并发访问，而Go的原生map不支持并发读写；Go在1.9的版本中新增了sync.Map的数据结构，两个map实现分离。

  1. 不会引发扩容的操作（查，改）使用read map，可能引发扩容操作（新增）使用 dirty map。

  2. 读写 read 并不需要加锁，而读或写 dirty 则需要加锁。

  3. 适合读多 append 少的场景，其实是读写和追加分离。

1. 介绍

1.1 数据结构

1.2 原理

sync.Map底层使用了两个原生map，一个叫read，用于正常读写；一个叫dirty，用于追加最新写入的key-value数据。

  • 追加的时候，在dirty增加数据，然后给 read 的 amended置为 true。
  • 如果misses增加，=len(dirty)，这时候把dirty 提升。然后重置 amended，和misses。

可概括为：

  1. 通过 read 和 dirty 实现数据的读写和追加分离，读写的数据存在只读字段 read 上，将最新append的数据则存在 dirty 字段上。
  2. 读取时会先查询 read，不存在再查询 dirty，追加时则只写入 dirty。
  3. 读取 read 并不需要加锁，而读或写 dirty 则需要加锁。
  4. 另外有 misses 字段来统计 read 被穿透的次数（被穿透指需要读 dirty 的情况），超过一定次数则将 dirty 数据更新到 read 中（触发条件：misses=len(dirty)）

  • 删除逻辑

    1. 正常删除，把底层的指针置为空。

    2. 追加后删除，这时候dirty还没有提升。提升后把删除的指针置为 expunged(生僻的删除)。

2.1 load 操作

因为写操作仅针对dirty（负责写操作的map），所以dirty是包含read的，最新且全量的数据。

2.2 store 操作

2.3 delete 操作

通过阅读源码我们发现sync.Map是通过冗余的两个数据结构(read、dirty),实现性能的提升。

为了提升性能，load、delete、store等操作尽量使用只读的read；为了提高read的key击中概率，采用动态调整，将dirty数据提升为read；对于数据的删除，采用延迟标记删除法，只有在提升dirty的时候才删除。

3. 头脑风暴

  • 底层使用了两个原生map，不会引发扩容的操作（查，改）使用read map，可能引发扩容操作（新增）使用 dirty map。
  • 两个原生map，其实是读写和追加分离，读取 read 并不需要加锁，而读或写 dirty 则需要加锁。
